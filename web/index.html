Local-only URL shortener

Encodes the URL into the URL — no server, no database, no lookup. Every shortened link is a self-contained, lossless, decodable string.

How it works

Every link this page produces is fully self-contained. The decoded URL is reconstructed entirely from the encoded string — there's no database, no lookup table on a server. Open the HTML file from disk and it still works.

The trick is to pack the original URL through a pipeline of tricks, each one shaving bits from a different angle:

1. Domain prefix table v5 — 1 byte saved per known site

85 hand-picked URL prefixes (https://www.youtube.com/watch?v=, https://github.com/, etc.) are shared by encoder and decoder. The longest matching prefix becomes a 1-byte index; 0xFF means "no prefix matched, raw URL follows." Because both ends agree on the table, the index alone reconstructs ~30 bytes of URL.

2. Deflate with a pre-shared dictionary v4 — biggest single lever

Plain deflate doesn't help on a 75-byte URL — its LZ77 window starts empty, so there's nothing to back-reference. We seed the window with a hand-curated 1.5 KB dictionary of common URL fragments (?utm_source=&utm_medium=, /wiki/, common TLDs, etc.). Hot strings live near the end of the dictionary because shorter LZ77 distances encode in fewer Huffman bits.

Both encoder and decoder load the same dictionary at init time, so it doesn't take a single byte of wire space.

3. Denser output alphabet v7 / v8 — bits per visible character

Base64 packs 6 bits into one ASCII character. Two ways to do better:

  • basE91: 91 ASCII chars × variable-length packing, ~6.5 bits/char. About 8% denser than base64. Output stays ASCII (lowest UTF-8 byte cost). All 91 chars are URL-fragment-safe under WHATWG.
  • Base32768: 32,768 carefully picked BMP code points (CJK Unified, CJK Extension A, Hangul Syllables — all 3 UTF-8 bytes, none surrogates, RTL, or formatting). Each visible character carries 15 bits. Visible length drops ~2.5× vs base64.

Trade-off: base32768 doubles UTF-8 bytes on the wire (each "char" is 3 bytes). The visible URL is short; the transmitted URL is roughly the same size as the original. Pick the alphabet to match what you're optimising — readability or wire bytes.

4. Per-URL mode picker v9 — pick whichever wins

For very short or random URLs (http://192.168.1.1:8080/admin), compression has no signal to work with — deflate's overhead exceeds any saving. The encoder tries all candidates, picks the smallest, and stamps a 1-character mode marker so the decoder knows which dispatch to use:

  • \ at position 0 → raw passthrough (rest is the URL itself)
  • ASCII printable at position 0 → basE91 + dict-deflate
  • CJK / Hangul (≥ U+3400) at position 0 → base32768 + dict-deflate

What didn't work

Brotli (with its 120 KB built-in static dictionary, designed for HTML) lost to our 1.5 KB URL-tuned dictionary on URL-shaped input — its dictionary is full of English words and HTML tags that don't appear in URLs, and its per-stream overhead is heavier than deflate's on ~75-byte inputs.
