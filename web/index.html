How it works

Every short link this page produces is fully self-contained. The encoded URL goes after a #, and URL fragments never leave the browser — even if this page were hosted on a server, the server would not see the encoded data. Open the HTML file from disk and the shortener still works.

The pipeline that turns a 75-byte URL into ~20 visible characters:

1. Domain prefix table v5 — replace 30 bytes with one byte

85 hand-picked URL prefixes (https://www.youtube.com/watch?v=, https://github.com/, etc.) are baked into both encoder and decoder. The longest matching prefix becomes a single byte index; 0xFF means "no prefix matched, raw URL follows." Because both ends already share the table, the index alone reconstructs ~30 bytes of URL.

2. Deflate with a pre-shared dictionary v4 — biggest single lever

Plain deflate doesn't help on a 75-byte URL — its LZ77 window starts empty, so there's nothing to back-reference. We seed the window with a hand-curated 1.5 KB dictionary of common URL fragments (?utm_source=&utm_medium=, /wiki/, common TLDs, etc.). Hot strings live near the end of the dictionary because shorter LZ77 distances encode in fewer Huffman bits.

Both encoder and decoder load the same dictionary at init time, so it doesn't take a single byte of wire space.

3. Denser output alphabet v7 / v8

Base64 packs 6 bits into one ASCII character. We do better with two different alphabets, picked by what you're optimising:

  • basE91: 91 ASCII chars, ~6.5 bits/char (variable 13/14-bit packing). About 8% denser than base64. Output stays ASCII (cheapest UTF-8 bytes). All 91 chars are URL-fragment-safe under WHATWG.
  • Base32768: 32,768 carefully picked BMP code points (CJK Unified, CJK Extension A, Hangul Syllables — all 3 UTF-8 bytes, no surrogates / RTL / formatting chars). Each visible character carries 15 bits. Visible length drops ~2.5× vs base64.

Trade-off: base32768 makes the URL look short, but each char is 3 UTF-8 bytes — wire size stays roughly equal to the original. basE91 saves bytes on the wire but loses the visual compactness. Pick whichever matches what you're optimising.

4. Structural preprocessor — pack digit/hex runs v10 — general-purpose

Deflate can't compress high-entropy character runs (commit SHAs, content hashes, large integer IDs), but still pays Huffman literal cost per character. So before deflate sees the URL, we scan for runs of pure digits (≥6 chars) or pure lowercase hex (≥8 chars, with at least one a-f) and replace each with a binary token: 0x01 LL … for digits, 0x02 LL … for hex. Markers are byte values below 0x20 — never valid in URL strings, so unambiguous.

A 19-character Twitter status ID packs from 19 ASCII bytes to 8; a 40-char Git commit SHA packs to 20 bytes. Crucially this is general structural knowledge about URLs — not pattern matching on specific sites — so it helps any URL containing such runs whether we've heard of the host or not.

5. Free dispatch — 0 bits of mode marker v9

The two alphabets occupy disjoint Unicode ranges: basE91 output is ASCII printable (U+0021…U+007E), base32768 output is CJK / Hangul (≥ U+3400). The decoder just inspects the first character and dispatches: 

if (firstChar.codepoint >= 0x3400) decodeBase32768(s);
else                                  decodeBasE91(s);

No marker char, no marker byte, no length prefix — the alphabet itself is the signal. Earlier drafts used a \ marker for an optional "raw" mode; dropping that was free density on every link.

What didn't work

Brotli (with its 120 KB built-in static dictionary, designed for HTML) lost to our 1.5 KB URL-tuned dictionary on URL-shaped input — its dictionary is full of English words and HTML tags that don't appear in URLs, and its per-stream overhead is heavier than deflate's on ~75-byte inputs.

URL grammar decomposition (parse into scheme · subdomain · base · TLD · path, encode each piece with its own table) also lost — by 0.01%, basically nothing. The reason: a hand-curated prefix table reduces a known (scheme, host) pair to a 1-byte index. Grammar decomposition needs ~10 bytes of structural overhead (marker + scheme + subdomain + base length + TLD index + …) to describe the same thing. With both modes available, grammar mode only wins for the ~4% of URLs that hit only the scheme-level fallback prefix — and even on those, the savings are 3-4 bytes per URL. The simpler mechanism (a static table + deflate-with-dict) is already doing all the work.
