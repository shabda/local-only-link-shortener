How it works

Every short link this page produces is fully self-contained. The encoded URL goes after a #, and URL fragments never leave the browser — even if this page were hosted on a server, the server would not see the encoded data. Open the HTML file from disk and the shortener still works.

The pipeline that turns a 75-byte URL into ~20 visible characters:

1. Domain prefix table v5 — replace 30 bytes with one byte

85 hand-picked URL prefixes (https://www.youtube.com/watch?v=, https://github.com/, etc.) are baked into both encoder and decoder. The longest matching prefix becomes a single byte index; 0xFF means "no prefix matched, raw URL follows." Because both ends already share the table, the index alone reconstructs ~30 bytes of URL.

2. Deflate with a pre-shared dictionary v4 — biggest single lever

Plain deflate doesn't help on a 75-byte URL — its LZ77 window starts empty, so there's nothing to back-reference. We seed the window with a hand-curated 1.5 KB dictionary of common URL fragments (?utm_source=&utm_medium=, /wiki/, common TLDs, etc.). Hot strings live near the end of the dictionary because shorter LZ77 distances encode in fewer Huffman bits.

Both encoder and decoder load the same dictionary at init time, so it doesn't take a single byte of wire space.

3. Denser output alphabet v7 / v8

Base64 packs 6 bits into one ASCII character. We do better with two different alphabets, picked by what you're optimising:

  • basE91: 91 ASCII chars, ~6.5 bits/char (variable 13/14-bit packing). About 8% denser than base64. Output stays ASCII (cheapest UTF-8 bytes). All 91 chars are URL-fragment-safe under WHATWG.
  • Base32768: 32,768 carefully picked BMP code points (CJK Unified, CJK Extension A, Hangul Syllables — all 3 UTF-8 bytes, no surrogates / RTL / formatting chars). Each visible character carries 15 bits. Visible length drops ~2.5× vs base64.

Trade-off: base32768 makes the URL look short, but each char is 3 UTF-8 bytes — wire size stays roughly equal to the original. basE91 saves bytes on the wire but loses the visual compactness. Pick whichever matches what you're optimising.

4. Structural preprocessor — pack URL-grammar patterns v10 + v13

Deflate can't compress high-entropy character runs and still pays Huffman literal cost per character. Before deflate sees the URL, we scan for four URL-grammar patterns and replace each with a marker byte sequence:

  • 0x01 LL … — digit run (≥ 6 chars). 19-char Twitter status → 8 bytes.
  • 0x02 LL … — lowercase hex run (≥ 8 chars, ≥ 1 a-f). 40-char Git SHA → 20 bytes.
  • 0x03 …3 bytes — /YYYY/MM/DD/ path. Year (14 bits) · month (4) · day (5) packed into 3 bytes. 12-char date path → 4 bytes. Used by every CMS / news platform since the early 2000s — fired on 10% of real URLs in the corpus.
  • 0x04 …16 bytes — RFC 4122 UUID (8-4-4-4-12 hex). 36 chars → 17 bytes.

Markers are byte values below 0x20 — never valid in URL strings, so unambiguous in the preprocessed stream.

Plus a canonicalisation step (RFC 3986 §6.2.2.2): %XX sequences for unreserved chars (A-Z a-z 0-9 - . _ ~) are decoded to their literal form. %7E → ~, %41 → A. Required by the RFC for canonical URI form — both representations route to the same resource.

None of these come from looking at any corpus. They're properties of URL syntax (RFC 3986/3987/4122) and universal internet conventions. They help on any URL that happens to contain them, whether we've heard of the host or not.

5. Free dispatch — 0 bits of mode marker v9

The two alphabets occupy disjoint Unicode ranges: basE91 output is ASCII printable (U+0021…U+007E), base32768 output is CJK / Hangul (≥ U+3400). The decoder just inspects the first character and dispatches: 

if (firstChar.codepoint >= 0x3400) decodeBase32768(s);
else                                  decodeBasE91(s);

No marker char, no marker byte, no length prefix — the alphabet itself is the signal. Earlier drafts used a \ marker for an optional "raw" mode; dropping that was free density on every link.

What didn't work

Brotli (with its 120 KB built-in static dictionary, designed for HTML) lost to our 1.5 KB URL-tuned dictionary on URL-shaped input — its dictionary is full of English words and HTML tags that don't appear in URLs, and its per-stream overhead is heavier than deflate's on ~75-byte inputs.

URL grammar decomposition (parse into scheme · subdomain · base · TLD · path, encode each piece with its own table) also lost — by 0.01%, basically nothing. The reason: a hand-curated prefix table reduces a known (scheme, host) pair to a 1-byte index. Grammar decomposition needs ~10 bytes of structural overhead (marker + scheme + subdomain + base length + TLD index + …) to describe the same thing. With both modes available, grammar mode only wins for the ~4% of URLs that hit only the scheme-level fallback prefix — and even on those, the savings are 3-4 bytes per URL. The simpler mechanism (a static table + deflate-with-dict) is already doing all the work.

A universal-only dictionary (v12: drop all site-specific entries, keep only RFC + framework conventions like /api/v1/, ?utm_source=, .html) made things worse on real data, not better: 0.383 → 0.412 chars on the real corpus. The reasoning was "the site-specific entries are corpus-tuned." The data said: real internet traffic actually does hit popular sites a lot — even on a held-out corpus, the dict's github.com/ · youtube.com/watch?v= · wikipedia.org/wiki/ entries earn their bytes. So v13 keeps them and just adds the new structural packers on top.

Benchmark

Two corpora. Synthetic = 1,000 hand-templated URLs generated from common shapes (YouTube/GitHub/Wikipedia/news/…) — the development bench. Real = 4,313 actual URLs scraped from Hacker News + Reddit (top + best + new across 30 subs) — what users actually paste. Both averaged ~75 chars.

Lower ratio = shorter. The two columns tell different stories about each trick: corpus-tuned mechanisms win big on synthetic data and shrink on real, while structural ones hold up.

Two findings worth noticing in this table:

  • v3 → v4 reverses sign on real data. The hand-curated dictionary saves 0.38 ratio on synthetic but costs 0.20 on real: the dict's prefix entries (https://www.youtube.com/watch?v=, github.com/, …) match the synthetic generator's templates almost perfectly, but on real Hacker News / Reddit data — heavy on long-tail indie blogs, news sites, and substacks — those entries rarely apply. The prefix table is in the same boat (v5 saves 0.08 on synthetic, 0.03 on real). This is the price of any knowledge-based trick.
  • v8/v10 hold up. The base32768 alphabet (15 bits/visible-char vs base64's 6) is corpus-independent; it gives a ~2.6× visible-char shortening on real URLs vs ~3.2× on synthetic. The v10 preprocessor still finds digit/hex runs in real URLs and packs them — a ~1% extra win, smaller than the synthetic 3% but still positive on data the encoder has never seen.

So a fairer headline number is the real one: this page compresses real-world URLs to ~38% of their visible length, losslessly, with no server. The bigger 31% number on synthetic data is a development artefact — it tells you where the dict and prefix table are tuned to live.

The two α rows in the scoreboard answer "how much does each layer actually contribute?". With no compression at all, just swapping ASCII for base32768 already drops visible chars to 0.540 on real URLs — about 75% of the eventual win (0.540 → 0.379) comes from the alphabet alone. The deflate + dict + prefix-table stack adds another 0.15; the structural preprocessor adds the last ~0.01. Conversely, on UTF-8 wire bytes the alphabet by itself is a villain (1.619×), and the compression stack is what claws those bytes back.
